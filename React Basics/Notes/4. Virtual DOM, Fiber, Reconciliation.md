# Virtual DOM
The Virtual DOM is a lightweight , in memory representation of real DOM used by React to optimize UI rendering and performance. 
Instead of directly manipulating the browser's slow HTML structure, React updates JavaScript object blueprint first. 
## The Cycle
### Virtual DOM Generation: 
When state changes, React builds a brand new VDOM tree with updated UI. 
### Diffing
React uses highly optimized algorithms to compare the new VDOM tree with previous VDOM snapshot. 
### Merging
It then calculate the minimum number of changes needed and updates only those specific nodes in the real DOM.

# Reconciliation
The algorithm React uses to diff one tree with another to determine which parts need to be changed. Mainly it is between Virtual DOM and Browser DOM. Here update in UI is defined by "A change in the data used to render a React app. Usually the result of `setState`. Eventually results in a re-render."

Before React-16, React used the Stack Reconciler which did reconciliation synchronously using the built-in JavaScript call stack.
After React 16, it was switched to a better Fiber Reconciler. It does reconciliation asynchronously by breaking work down into independent "fiber" units.

Diffing is a specific heuristic algorithm used by React in the Reconciliation process to compare a node in the old tree with the node in the new tree. Fiber did change Diffing rules. It changed when and how that diffing work is executed. 
# Fiber
[Architecture of Fiber](https://github.com/acdlite/react-fiber-architecture)
## Problem
Before Fiber, React used the Stack Reconciler. 
- **Synchronous updates**: React processed UI updates in a single, continuous execution stack.
- **Main thread blocking**: Once rendering started, it could not be paused. If a component tree was deep and complex, it would block the main browser thread, causing lag, dropped frames, and frozen inputs
## Solution 
Fiber is a plain JS object that represents a virtual unit of work. . Instead of executing everything recursively as a call stack, React Fiber organizes components into a **singly-linked list of fiber nodes**. Each node holds metadata about a component's state, props, and its relationship to other DOM elements. 
## Core Features
- **Incremental Rendering**: 
  The ability to break rendering work into small chunks and spread them out over multiple frames. 
  
- **Work Pausing and Resuming**:
  React can pause a rendering job to handle something else, then resume it after the work has been done. 
  
- **Concurrency**:
  Multiple tasks can be processed at the same time without locking up the main UI thread. 
  
- **Task Prioritization**:
  React assigns high priority to urgent user interactions like typing and animations and assign low priority to updates like data fetching. 

## Key Points: 
- Different component types are assumed to generate substantially different trees. React will not attempt to diff them, but rather replace the old tree completely.
  
- Diffing of lists is performed using keys. Keys should be "stable, predictable, and unique."
